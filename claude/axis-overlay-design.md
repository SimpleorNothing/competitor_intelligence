# 중첩축(Overlay Axis) 설계 — A7 B2B 주거 솔루션

작성 2026-07-19 · 대상 LG전자 축 구조

## 배경

건설사 대상 '구독 + AI홈 + 로봇' 번들이 3개월 내 시퀀스로 진행됐다.

| 시점 | 사건 | 귀속 축 |
|---|---|---|
| '26.4 | GS건설 로봇친화형 아파트 MOU (클로이드) | A6 |
| '26.7.10 (발표 7.13) | GS건설 차세대 AI홈 공동개발 MOU — 류재철 CEO 참석, 씽큐온↔자이 단지인프라, 성수1지구부터 클로이드·자율주행 서빙배송 로봇 | A3 + A6 |
| '26.7.19 | 현대건설 압구정 재건축 2·3·5구역 조합원 7,000세대 가전 구독 | A1 |

같은 고객(건설사)에게 세 축의 산출물을 묶어 판다. 축 하나로는 담기지 않고, 그렇다고 새 축으로 빼면 A1·A3·A6에서 증거를 뽑아내야 해서 각 축의 궤도 판정이 망가진다.

## 결정

**A7은 배타적 분리축이 아니라 중첩축(overlay)으로 신설한다.**

- 기존 A1~A6: **전략 방향**으로 정의되는 축. 증거는 여기에 단일 귀속(`axisId`).
- A7: **고객군·채널**로 정의되는 축. 증거를 옮겨오지 않고 `tags`로 참조한다.

즉 `a or b`가 아니라 `a이면서 b`. 압구정 건은 **A1 증거이면서 동시에 A7 증거**다.

### 왜 이 구조여야 하나

증거를 A7로 옮기면 A1의 kpiSeries(구독 매출 궤도)에서 B2B 채널 기여분이 빠진다. 실제로는 같은 구독 매출인데 채널만 다른 것이므로, 궤도 판정 관점에서는 A1에 남아 있어야 맞다. 동시에 "건설사 채널이 얼마나 커지고 있나"라는 질문도 답해야 하므로 교차 조회가 필요하다. 두 요구를 동시에 만족시키는 유일한 구조가 중첩이다.

## 스키마 규약

### evidence item — `tags` 추가

```json
{
  "id": 101,
  "axisId": "lg-a1",          // 단일 귀속 유지 — 변경 없음
  "tags": ["B2B주거"],         // 신규. 중첩축 참조
  "event": "...",
  ...
}
```

- `tags`는 **선택 필드**. 없으면 기존과 동일하게 동작한다(하위 호환).
- 값은 중첩축 식별자. 현재 정의된 것은 `B2B주거` 하나.
- 한 증거가 여러 태그를 가질 수 있다.
- **`axisId`는 절대 중첩축 id로 바꾸지 않는다.** 중첩축은 `axisId`의 값 domain이 아니다.

### 중첩축 정의 (신설 시 strategies.json)

```json
{
  "id": "lg-a7",
  "code": "A7",
  "kind": "overlay",           // 신규 필드. 기본값 "axis"
  "tag": "B2B주거",             // 이 태그를 가진 증거를 모아 보여준다
  "title": "B2B 주거 솔루션",
  "spans": ["lg-a1", "lg-a3", "lg-a6"]
}
```

- `kind: "overlay"` 축은 **자체 kpiSeries·trajGrade를 갖지 않는다.** 궤도 판정은 원 축이 담당한다.
- 대신 `execStatus`와 채널 고유 KPI(누적 계약 세대수, 제휴 건설사 수 등)는 가질 수 있다.
- 보드 UI에서 A7 탭은 필터 뷰로 동작한다 — A1 탭에서 압구정 건이 사라지면 안 된다.

## 신설 트리거

`watchlist.json`의 `w-a1-b2b.splitTriggers`와 동기화된 판정 기준.

| id | 기준 | 현재 |
|---|---|---|
| t1 | 계약 세대수 누계 또는 B2B 주거 매출·수주 정량 공표 | 미충족 |
| t2 | 전담조직이 사업본부급 또는 CEO 직속 승격 (현 한국B2B그룹은 HS 산하 영업조직) | 미충족 |
| t3 | IR·주총·컨콜에서 독립 성장축으로 명시 | 미충족 |

**2개 이상 충족 시 A7 신설.**

A6 로봇축 신설 경로가 선례다 — 조직 신설 → CEO의 '로봇 원년' 선언 → 파일럿 라인 구축. B2B 주거는 아직 1단계도 오지 않았다.

## 잔여 작업

1. `/api/promote`가 `tags`를 통과시키도록 수정 — 현재 evidence item 생성 시 `tags`를 버린다.
2. 보드 UI에 태그 필터 추가 (A7 신설 전에도 `B2B주거` 교차 조회가 가능해야 함).
3. `evidence.json`은 90KB로 인라인 커밋 한계를 넘는다. 기존 증거 소급 태깅이 필요해지면 Claude Code 로컬 환경 필요. 단 현재 4건은 아직 evidence 미등재(인박스 단계)라 소급 대상 없음.

## 반론 기록

건설사 번들이 계속 커지면 "주거 솔루션"이 A1·A3·A6의 **하위 채널이 아니라 상위 개념**이 될 수 있다. 모듈러 주택 진출까지 겹치면 더 그렇다. 그 경우엔 축 하나 추가가 아니라 축 구조 재설계가 맞다. 현재는 정량 근거가 0이라 판단 시점이 아니며, 트리거 t1(정량 공표)이 충족되는 순간 이 반론을 다시 검토한다.
